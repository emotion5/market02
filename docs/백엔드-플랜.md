# MMM MARKET — 백엔드 구축 상세 플랜

방향: **① Next.js 통합(App Router `app/api`) + `server/` 레이어로 백엔드 고립.**
프론트(쇼핑몰 + 어드민)와 백엔드가 한 프로젝트(`market02`)에 있지만, 코드 레이어로
완전히 분리해 "나중에 `server/`만 떼어내 별도 서비스로 승격" 가능한 구조로 간다.

---

## 파트 A. 현재 데이터 구조 (교체 대상)

지금은 실제 DB가 없고, 아래 두 축으로 데이터가 존재한다.

### A-1. localStorage 저장소 (클라이언트, 사용자별 브라우저)

| 키 | shape | 정의 위치 | 대체 대상 |
|---|---|---|---|
| `market02-auth` | `{ email }` | `hooks/useAuth.tsx` | 진짜 세션(쿠키) |
| `market02-cart` | `CartItem[]` | `hooks/useCart.tsx` | (유지 가능) 선택적 DB 동기화 |
| `market02-quotes` | `SavedQuote[]` | `lib/quotes.ts` | `Quote` + `QuoteItem` 테이블 |
| `market02-orders` | `Order[]` | `lib/orders.ts` | `Order` + `OrderItem` 테이블 |
| `market02-addresses` | `Address[]` | `mypage/addresses` | `Address` 테이블 |
| `market02-profile` | `{ bizNo, company, manager, tel, email }` | `mypage/profile` | `User` + `BusinessProfile` |

### A-2. 서버에서 읽는 mock JSON (카탈로그)

| 파일 | 내용 | 대체 대상 |
|---|---|---|
| `data/mock-products.json` | 상품 130개 (id·name·category·price·variants·colors·image) | `Product`/`Variant`/`ProductColor`/`ProductImage` |
| `data/featured.json` | `{ categorySlug: [productId...] }` 큐레이션 | `Featured` 테이블 |
| `lib/constants.ts` `CATEGORIES` | 6개 카테고리 | `Category` 테이블 |
| `public/images/products/**` + 파일명 규칙 스캔 | 상세 갤러리 이미지 | `ProductImage.url` |

카테고리: `hardware`(하드웨어) · `panel`(가구재) · `pressbevel`(바닥재) · `antipress`(장판) · `bathmatch`(위생) · `stablecore`(건자재).

### A-3. ★ 이미 있는 교체 이음새 (핵심)

페이지 코드를 거의 안 건드리고 백엔드로 바꿀 수 있는 이유:

- `lib/data.ts` — 함수 내부에 `// ★ 백엔드 교체 지점` 주석. `getProducts()` 등 시그니처 유지한 채 내부만 API 호출로 교체.
- `lib/orders.ts` / `lib/quotes.ts` — `getOrders()`·`saveOrder()`·`getQuotes()`·`saveQuote()` 등의 함수만 API 호출로 교체.
- `hooks/useAuth.tsx` — `login()`이 서버 검증으로 대체된다고 주석에 이미 명시.

**전략: 이 함수들의 "속"만 바꾸고 페이지는 그대로 둔다.**

### A-4. 도메인 규칙 (스키마에 반영해야 할 불변식)

1. **회원 2종**: 개인(즉시 가입) / 사업자(사업자등록증 업로드 → 관리자 승인 → **회원도매가** 적용).
2. **이메일 = 로그인 아이디이자 계정 PK.** 사업자등록번호는 아이디가 아니라 확인 자료.
3. **견적서·주문은 발행/주문 시점 스냅샷으로 고정(불변).** 이후 상품 가격이 바뀌어도 과거 문서 금액 불변 — 회계·세금 정합성.
4. **견적번호** `Q-YYYYMMDD-HHMMSS`, **유효기간 14일**. **주문번호** `YYYYMMDD-HHMMSS`.
5. **금액 규칙**: 표시가 = 부가세 포함. `공급가액 = round(합계/1.1)`, `부가세 = 합계 − 공급가액`.
6. **주문 상태 5단계**: 입금대기 → 입금확인 → 배송준비 → 배송중 → 배송완료. (지금은 경과시간 기반 mock `deriveStatus` → 서버가 관리하는 실제 상태로 대체)
7. **세금계산서**: 주문 시 신청, 입금확인 이후 발행완료.

## 파트 B. 목표 DB 스키마 (PostgreSQL + Prisma)

> 원칙: 카탈로그(가변) 테이블과 문서(불변 스냅샷) 테이블을 명확히 분리.
> 상품 id는 기존 `"bathmatch-0001"` 형태 문자열 키를 그대로 유지(이미지 경로·featured 호환).

### 회원 / 인증
```
User            id, email(uniq), passwordHash, name, tel,
                type(PERSONAL|BUSINESS), status(ACTIVE|PENDING|REJECTED|WITHDRAWN),
                role(CUSTOMER|ADMIN), createdAt, withdrawnAt?
BusinessProfile userId(uniq FK), bizNo, company, owner?,
                licenseFileUrl, approvedAt?, approvedBy?(admin userId), rejectReason?
Address         id, userId(FK), label, recipient, tel, address, isDefault
```
- 회원도매가는 `User.type == BUSINESS && status == ACTIVE`일 때 적용(파트 B 가격 참고).
- 세션은 테이블 대신 httpOnly 쿠키 JWT로 처리(파트 G).

### 카탈로그 (가변)
```
Category      slug(PK), nameKo, nameEn, sortOrder
Product       id(PK, 문자열), name, categorySlug(FK), summary?, description,
              repImage, price(대표가), sortOrder, isActive, createdAt
Variant       id(PK, 문자열), productId(FK), name, price(소비자가),
              wholesalePrice?(회원도매가), sortOrder
ProductColor  id, productId(FK), hex, sortOrder
ProductImage  id, productId(FK), url, sortOrder   // 파일명 규칙 스캔 대체
Featured      categorySlug(FK), productId(FK), sortOrder   // featured.json 대체
```

### 견적서 (불변 스냅샷)
```
Quote      id, number(uniq), userId(FK), issuedAt, validUntil,
           customerCompany, customerContactName, customerContactTel,
           supply, vat, total, status(ISSUED|EXPIRED)
QuoteItem  id, quoteId(FK), productId, productName, variantId, variantName,
           color?, unitPrice, quantity, image   // 전부 발행 시점 값 복제
```

### 주문 / 결제 / 세금계산서 (불변 스냅샷)
```
Order        id, orderNo(uniq), userId(FK), createdAt,
             ordererName, ordererTel, ordererAddress, ordererMemo?,
             depositorName, paymentMethod(BANK_TRANSFER|PG),
             status(PENDING|PAID|PREPARING|SHIPPING|DELIVERED),
             supply, vat, shippingFee, total,
             courier?, trackingNumber?
OrderItem    id, orderId(FK), productId, productName, variantId, variantName,
             color?, unitPrice, quantity, image   // 주문 시점 값 복제
Payment      id, orderId(FK), provider(PORTONE|TOSS|NONE), method?, pgTid?,
             status(READY|PAID|CANCELLED|FAILED), amount, paidAt?,
             receiptUrl?, raw(json)   // PG 단계에서 사용
TaxInvoice   id, orderId(uniq FK), requested, bizNo?, company?,
             status(NONE|PENDING|ISSUED), issuedAt?, ntsRef?
```

> 스냅샷 원칙: `QuoteItem`/`OrderItem`은 `Product`를 FK로 참조만 하지 말고
> 이름·단가·이미지를 **값으로 복제**해 저장한다(상품이 바뀌거나 삭제돼도 문서 불변).

## 파트 C. 폴더 구조 & 레이어 규칙 (① + server 고립)

```
market02/
├─ app/
│  ├─ (기존 쇼핑몰 페이지)         ← 프론트: 화면만. DB/서비스 직접 호출 금지
│  ├─ admin/                     ← 어드민 화면 (role=ADMIN만 접근)
│  │   ├─ members/  (사업자 승인)
│  │   ├─ products/ (상품 CRUD)
│  │   ├─ orders/   (주문상태·운송장)
│  │   └─ tax-invoices/
│  └─ api/                       ← 경계(HTTP): 인증확인 → 서비스 호출 → 응답. 로직 금지
│     ├─ auth/  products/  quotes/  orders/  addresses/
│     ├─ admin/
│     └─ webhooks/pg/            ← PG 결제 콜백
├─ server/                       ← ★ 백엔드 본체 (고립 영역)
│  ├─ db.ts                      (PrismaClient 싱글턴)
│  ├─ auth/      (service, session, password)
│  ├─ catalog/   (service, repo)
│  ├─ quotes/    (service, repo)
│  ├─ orders/    (service, repo)
│  ├─ payments/  (service, providers/portone.ts …)  ← PG 추상화
│  └─ admin/     (service)
├─ lib/                          ← 프론트·백 공용 (types.ts, zod 스키마)
├─ prisma/schema.prisma
└─ middleware.ts                 ← /admin, /mypage 접근 보호
```

**불변 규칙 (이걸 지켜야 "고립"이 유지됨):**
1. `app/**` 화면은 **절대** Prisma·`server/**`를 직접 import 하지 않는다. 오직 `/api` 또는 기존 `lib/data.ts` 같은 클라이언트 게이트웨이를 통해서만.
2. `app/api/**/route.ts`는 **얇게**: 입력 zod 검증 → 세션 확인 → `server/*/service` 호출 → 응답. 비즈니스 로직·SQL 없음.
3. 비즈니스 로직·DB 접근은 **전부** `server/**`. Prisma import는 `server/**`에서만.
4. `server/**`는 Next·React를 import 하지 않는다(=> 그대로 NestJS로 이식 가능).

## 파트 E. 마이그레이션 순서 (localStorage → API, 위험 낮은 순)

각 단계는 **기존 `lib/*.ts` 함수의 내부만 fetch로 교체** → 페이지 무수정이 목표.

- **Phase 0 — 인프라**: Postgres 연결, Prisma 스키마·마이그레이션, `server/db.ts`, zod, 세션(쿠키) 뼈대, `middleware.ts`, `server/` 레이어 규칙 셋업.
- **Phase 1 — 카탈로그(읽기, 최저위험)**: mock JSON → DB 시드. `lib/data.ts` 내부를 `server/catalog` 호출로 교체. 어드민 상품 CRUD. `ProductImage`로 파일명 스캔 제거.
- **Phase 2 — 인증/회원**: 개인·사업자 가입, 로그인/세션, 사업자등록증 업로드(오브젝트 스토리지), **사업자 승인 어드민**, 프로필. `useAuth` 실제 세션으로 교체.
- **Phase 3 — 배송지**: 사용자별 CRUD로 교체(간단).
- **Phase 4 — 견적서**: 발행/목록/열람을 API+스냅샷으로. `lib/quotes.ts` 저장부 교체.
- **Phase 5 — 주문/배송**: 체크아웃 주문 생성(스냅샷), 내역/상세. `deriveStatus` mock → 서버 실제 상태·어드민 상태변경·운송장.
- **Phase 6 — 세금계산서**: 신청 저장·발행완료 처리(전자세금계산서 연동은 옵션).
- **Phase 7 — PG 결제**: 무통장입금과 병행. `payments` 추상화 위에 PortOne(또는 선택 PG) 구현, 결제 준비 + 웹훅 검증 → 주문 PAID 전이.

각 Phase 완료 시 `/dev/seed`(현재 localStorage 시드용) 의존 제거, 배포 시 `/dev` 라우트 제외.

## 파트 F. PG 결제 흐름 (Phase 7 상세)

```
1) 프론트: /checkout 에서 주문 생성(POST /api/orders, status=PENDING)
2) 프론트: POST /api/payments/prepare  → 서버가 금액을 DB 기준으로 확정,
           PG 주문키 발급, 결제창 파라미터 반환 (금액을 클라가 못 바꾸게)
3) 프론트: PG 결제창(SDK) 호출 → 사용자 결제
4) PG → 서버: POST /api/webhooks/pg (서버-서버). 서명·금액 재검증 후
           Payment=PAID, Order.status=PAID 로 전이 (여기가 신뢰 지점)
5) 프론트: 완료 화면. 상태는 서버 조회 결과만 신뢰(결제창 반환값 신뢰 금지)
```
- **금액·재고 검증은 100% 서버.** 결제창/리다이렉트 반환값은 표시용, 확정은 웹훅.
- `payments/providers/*`로 PG를 추상화 → PortOne/토스 교체·병행 가능.
- 무통장입금 주문은 `paymentMethod=BANK_TRANSFER`로 남고, 어드민 입금확인이 PAID 전이.

## 파트 G. 기술 스택 (확정 제안)

| 영역 | 선택 | 이유 |
|---|---|---|
| DB | **PostgreSQL(관리형)** | 관계·트랜잭션·정합성. 커머스 표준 |
| ORM | **Prisma** | 타입 안전, 마이그레이션, 스냅샷 모델링 명확 |
| 검증 | **zod** | `lib/`에 두고 프론트·API 공용 |
| 인증 | **bcrypt/argon2 + jose(httpOnly 쿠키 JWT)** | 사업자 승인 게이팅 등 커스텀 흐름에 유연. (대안: Auth.js Credentials) |
| 파일 | **오브젝트 스토리지(S3 호환/R2/Supabase Storage)** | 사업자등록증·상품 이미지. DB엔 URL만 |
| PG | **PortOne(권장) — 추상화 뒤에 구현** | 여러 PG 스위칭, B2B 무통장+카드 병행 |
| 배포 | 프론트+API 한 배포. DB는 관리형 | ①구조에 맞음 (환경은 미정 → 추후 확정) |

## 미정/결정 필요 항목

1. **PG사**: PortOne(권장) vs 토스 직접 vs 당장은 무통장만.
2. **배포/DB 호스팅**: 국내 리전(네이버/NHN/카페24) vs Vercel+관리형 Postgres vs AWS.
3. **오브젝트 스토리지** 선택(위 호스팅과 함께 결정).
4. **인증 라이브러리**: 커스텀(jose) vs Auth.js — 커스텀 권장하나 취향 확인.
5. 장바구니(`market02-cart`)를 로그인 사용자 한정 DB 동기화할지(선택).

> 이 문서는 계획서다. 다음 단계로 Phase 0(인프라·`server/` 뼈대 + Prisma 스키마 코드화)부터 착수한다.
