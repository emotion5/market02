# MMM MARKET — 어드민 설계도

쇼핑몰(손님 화면)의 각 도메인에는 **관리(어드민) 짝**이 있다. 여기에 운영 전용 기능(대시보드·설정)을 더한 것이 어드민 전체다. 공통 토대(레이아웃·UI 키트·ADMIN 가드)는 이미 구축됨.

> 원칙: 어드민 화면은 대응하는 **도메인 DB가 있어야** 만들 수 있다. 그래서 백엔드 phase 진행과 짝을 이룬다. (예: 주문 관리는 주문 DB=Phase 5 이후)

## 모듈 한눈에

| 모듈 | 경로 | 목적 · 주요 기능 | 선행(필요 DB) | 상태 |
|---|---|---|---|---|
| **대시보드** | `/admin` | 운영 요약: 승인대기·입금대기·배송대기·세금계산서 대기·신규가입 카운트 → 각 화면으로 | 여러 도메인(점진적으로 채움) | ⬜ |
| **회원 관리** | `/admin/members` | 사업자 승인/반려, 전체 회원 목록·검색(이메일/상호/번호), 상세, 상태(활성·정지·탈퇴), 등급(회원도매가) | 회원 DB ✅ | ✅ 완료 (목록·검색·상세·승인·정지·등급) |
| **상품 관리** | `/admin/products` | 상품 CRUD·활성화, 옵션·색상·이미지, 대표가·**회원도매가** 설정, 카테고리, 홈 큐레이션(featured) 편성 | 카탈로그 DB ✅ | ✅ 완료 (CRUD·옵션·이미지·홈 편성) |
| **주문 관리** | `/admin/orders` | 주문 목록(상태·기간·검색), 상세(스냅샷), 무통장 입금확인, 상태 전이(입금대기→…→배송완료), 운송장·택배사, 취소/환불 | 주문 DB (Phase 5) | ⬜ 선행 필요 |
| **견적서 관리** | `/admin/quotes` | 발행 견적 조회(고객·번호·기간), 상세, 유효기간(대부분 조회 위주) | 견적 DB (Phase 4) | ⬜ 선행 필요 |
| **세금계산서 관리** | `/admin/tax-invoices` | 신청 건 목록, 발행완료 처리, 발행 내역 | 주문·세금계산서 DB (Phase 5~6) | ⬜ 선행 필요 |
| **사이트 설정** | `/admin/settings` | 공급자 정보·입금계좌·견적 유효기간·고객센터 연락처 — 지금 `lib/constants` 에 하드코딩된 값을 DB화해 편집 | 설정 테이블(신규) | ⬜ |
| **관리자·권한** | `/admin/staff` | 복수 관리자, 역할 세분화(현재는 단일 ADMIN) | 회원 DB ✅ | ⬜ 후순위 |

## 손님 화면 ↔ 어드민 짝 (매핑)

| 손님 | 어드민 |
|---|---|
| 상품 탐색·검색·상세 (`data/mock-*` → DB) | 상품 관리 |
| 회원가입·로그인·마이페이지/회원정보 | 회원 관리 |
| 견적서 발행·조회 | 견적서 관리 |
| 주문/결제·배송조회 | 주문 관리 |
| 세금계산서 내역 | 세금계산서 관리 |
| (하드코딩된 공급자·계좌·유효기간) | 사이트 설정 |

## 권장 빌드 순서 (백엔드 phase와 맞물림)

1. **상품 관리** — 카탈로그 DB가 이미 있어 **지금 바로 가능**. JSON 손편집 제거 + 회원도매가를 어드민에서 설정. 다른 어드민 화면의 본보기(목록·폼·이미지 업로드 패턴).
2. **회원 관리 확장** — 승인 외 전체 회원 목록/검색/상세/상태. (회원 DB 완료)
3. **사이트 설정** — 작은 설정 테이블 하나로 하드코딩 값 제거. (독립적, 아무 때나)
4. **대시보드** — 위 지표들이 생기는 대로 요약 화면.
5. **주문 관리** — Phase 5(주문 DB) 이후.
6. **견적서 관리** — Phase 4(견적 DB) 이후.
7. **세금계산서 관리** — Phase 6 이후.
8. **관리자·권한** — 필요 시.

## 공통 원칙 (모든 어드민 화면)

- **UI 키트 재사용**: `app/admin/admin.module.css`(테이블·카드·버튼·뱃지) + `StatusPill`·`AdminNav` + 앞으로 추가할 `DataTable`·`Toolbar`(검색/필터)·`Pagination`·`Modal`.
- **접근 가드 2중**: `proxy.ts`(낙관적) + 레이아웃/`getAdmin`(서버 확정). 모든 어드민 API는 `getAdmin` 재확인.
- **감사 흔적**: 누가 언제 처리했는지 기록(예: 승인의 `approvedById`/`approvedAt`). 주문 상태변경 등에도 확장.
- **목록 공통 패턴**: 검색 + 필터 + 정렬 + 페이지네이션 (상품·주문·회원이 공유).
- **CSS Modules 유지** — 유틸리티 프레임워크 미도입, 공유 키트로 반복 최소화.
- **쇼핑몰과 UI 완전 분리**, 데이터·도메인 로직(`server/**`)은 공유(단일 진실).

## 미정/결정 필요

- 이미지·파일 업로드 스토리지(Vercel Blob 등) — 상품 이미지·사업자등록증 공통으로 필요. 상품 관리 착수 시 함께 결정.
- 재고 관리 도입 여부(현재 스키마에 재고 없음 — 필요하면 Variant에 stock 추가).
- 관리자 여러 명·역할 세분화 시점.
